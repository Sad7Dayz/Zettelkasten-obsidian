## ✅ 요약: 가장 좋은 문서 사용 흐름

```
요구사항 정리 → 구조 설계 → DB/Flowchart → 스프린트 계획 → 칸반 → 개발 → 회고 → 문서 업데이트
```

이렇게 하면 문서가 흐름에 따라 유기적으로 연결되고,  
**스프린트 기반 개발**이 훨씬 체계적으로 돌아갑니다.
[Task DashBoard](https://www.notion.so/2a60e887d435807892a5ea5f4f376752?v=2a60e887d43580a98977000c46ea4ecf&source=copy_link)

## 📌 문서별 작성 시점 & 방식

| 순서  | 문서명                         | 시점           | 목적                 | 작성 방식 / 툴                             |
| --- | --------------------------- | ------------ | ------------------ | ------------------------------------- |
| 1️⃣ | **요구사항 명세서**                | 프로젝트 시작 전    | 고객/기획자와 기능 범위 정리   | Confluence, Notion, Google Docs 등     |
| 2️⃣ | **UML** (Use Case, Class 등) | 요구사항 정리 후    | 시스템 구조 설계 및 역할 파악  | Draw.io, Lucidchart, StarUML          |
| 3️⃣ | **Flowchart**               | 기능 흐름 설계 시   | 기능의 흐름 및 조건 로직 시각화 | Whimsical, Lucidchart 등               |
| 4️⃣ | **테이블 명세서**                 | 백엔드 설계 초기    | DB 구조 명세 및 컬럼 설명   | Excel, Notion Table, DB Designer 등    |
| 5️⃣ | **스프린트 계획서**                | 각 스프린트 시작 전  | 이번 스프린트 목표 및 일정 정리 | Notion, Confluence, Jira              |
| 6️⃣ | **칸반 보드**                   | 개발 중 (매일 사용) | 작업 흐름 관리 및 병목 제거   | Jira, Trello, Notion, GitHub Projects |
| 🔁  | **회고 정리 + 문서 갱신**           | 스프린트 종료 시    | 문서 최신화 + 리팩토링      | Notion, Confluence 업데이트               |

## 🔁 예시 흐름 (스프린트 기준)

### 🏁 [프로젝트 착수 시]

- ① 요구사항 명세서 작성 (고객 or PO)
    
- ② UML로 구조 파악
    
- ③ Flowchart로 흐름 확인
    
- ④ DB 설계 및 테이블 명세 작성(ERD)


### 🏃 [스프린트 시작 전]

- ⑤ 스프린트 계획 문서 작성
    
- ⑥ Jira 등에 칸반 보드 생성
    
- → 이 단계부터 **Jira Issue + 칸반 + 문서**가 링크로 엮임
    

### 💻 [개발 중]

- 칸반으로 작업 흐름 관리
    
- 문서 중간중간 필요시 업데이트 (예: API 명세, 테이블 수정, 회의록)
    
- [[코드리뷰| 코드리뷰]]
### ✅ [스프린트 종료 후]

- 회고 (Retrospective)
    
- 필요 시 문서 정리 및 다음 스프린트로 피드백 반영
    
-  정적 분석
    
-  Test(유닛, 통합, E2E)

## 🧩 연결 예시

- 요구사항 명세서 → **UML Use Case 다이어그램** 링크
    
- UML → **DB 구조 설계 / API 흐름 정의**
    
- API Flowchart → **스프린트 Task**로 분해
    
- 테이블 명세서 → **Jira 백엔드 티켓 Description**에 첨부
    
- 스프린트 계획서 → **칸반 보드 링크**, 각 Task → 문서로 연결
