## 제3장: 코드와 함께 춤을

- 🔹 기술 부채를 상환하는 바업

	- 부채의 위험과 비용을 기술

	- 대안에 대해 논의
	- 트레이드오프

	- 변경지점을 확인
	- 테스트할 지점을 확인
	- 의존성을 나눔
	- 테스트작성
	- 변경을 적용하고 리펙토링
- 🔹 버전 제어 시스템의 권장 기법을 활용.
	- https://cbea.ms/git-commit/
- 🔹 되도록 검증된 기술 사용
- 🔹 업스트림 커밋없이 포크만 하는것은 금물
-       참고 : 조너선 보카라 The Legacy code Programmer's Toolbox, 마탄 파울러 리펙토링2판, 브레드 브룩스  맨먼스미신
## 제4장: 운영 환경을 고려한 코드 작성
- 🔹 null값 사용 피하자
- 🔹 불변 변수 사용
- 🔹 입력값 검사
- 🔹 예외 사용(구체적 사용), 예외를 일찍 던지고 나중에 처리하자
- 🔹 리소스 해제
- 🔹 로그레벨
	- Trace : 상세한 내용 출력 레벨
	- Debug : 프로덕션 상황에서 문제가 발생 했을때 적합한 레벨
	- Info : 애플리케이션 상태
	- warn : 잠재적 문제가  될 만한 상황 출력 레벨
	- error : 살펴 봐야 할 레벨
	-  Fatal : 가장 위험한 수준 출력 레벨
- 🔹 로그 원자적 작성
- 🔹 로그 수집 신속
- 🔹 민감 데이터는 로그 기록 말자.
- 🔹 모든것을 측정하자
	- 리소스 풀
	- 캐시
	- 데이터구조
	- cpu 집약적 작업
	- io 집약적 작업
	- 데이터 크기
	- 예외와 에러
	- 원격 요청 및 응답
- 🔹 분산환경에서 더욱 중요해진 추적(RPC 호출)
- 🔹 설정으로 런타임 동작 소정
	- INI, JSON, YAML처럼 일반텍스트
	- 환경 변수
	- 명령줄 플래그
	- 커스텀 도메인 전용 언어(DSL)
	- 애플리케이션 구현에 사용된 언어
- 🔹 로그에 기록하고 검증
- 🔹 관련된 설정을 그룹화
- 🔹 배포된 설정은 변경금지
- 🔹 도구가 운영의 성패를 결정
- 🔹 개발자의 필수 체크리스트
	- 런타임 에러보다 컴파일 타임에러를 검출
	- 가능하면 불변 변수를 사용
	- 입출력 검증
	- owasp 취약점 숙지
	- 버그 확인 도구는 물론 타입 또는 타입 힌트도 사용
	- 예외 발생시 리소스를 해제
	- 애플리케이션 설정 추가
	- 모든 설정을 검증하고 로그에 기록
-       참고  : 스티브 맥코넬 code complete, 로버트 C.마틴 Clean code, 아마존 빌더 라이브러리, SRE를 위한 시스템 설계와 구축, 사이트 신뢰성 엔지니어링
## 제5장: 피할수 없는 코드 의존성의 관리

	- 뭔가 잘못됐을때 의존성을 수정하는것은 얼마나 쉬운가?
	- 의존성은 얼마나 성숙해 있는가?
- 🔹 의존성 격리
- 🔹 의존성은 신중하게 추가
- 🔹 버전을 고정
- 🔹 의존성 범위를 좁히자.
- 🔹 순환 의존성 주의
- 🔹 의존성 범위를 좁히자.

## 제7장: 올바로 주고 받는 코드리뷰
- 🔹 코드리뷰는 왜 필요한가
- 🔹 코드리뷰를 제대로 받는 방법
- 🔹 리뷰초안이 있으면 위험을 낮출수 있다.
	- 변경사항이 작업중(WIP)
	- 코드리뷰 제목 Draft or WIP(ex. Draft PR)

- 🔹 테스트 실행을 위한 리뷰 제출은 금물
- 🔹 코드 변경사항이 많을때는 좀 더 면밀하게
	- 설명 할 시간을 갖자.
	- 미리 배포해서 회의 참석
- 🔹 자신의 코드에 너무 집착하지 말자.
- 🔹 공감력을 갖되 무례함은 참지 말자.
- 🔹 주도적으로 행동
- 🔹 코드 리뷰를 제대로 해주는 방법
	- 리뷰요청을 선별
	- 리뷰를 위한 시간을 마련
	- 코드 변경사항을 이해
	- 포괄적인 피드백을 제시
	- 좋은 점은 인정하자.
	- 이슈, 제안, 사소한 흠결은 잘 구분하자. (nit: 공백이 두번 입력됐어요.)
	- 대충대충 리뷰는 금물
	- 웹기반 리뷰 도구에만 의존하지말자.
	- 테스트 리뷰도 잊지말자.
	- 어떻게든 결론을 맺어야한다.
 -       참고  : 구글의 코드리뷰 가이드 https://google.github.io/eng-practices/
## 제8장: 고객앞으로! 소프트웨어 전달
- 🔹 소프트웨어 전달의 4가지 단계
	 (산출물을 패키징하는것을 릴리스, 산출물을 다운로드 가능하는 상태로 만드는것은 발행)
	- 빌드
	- 릴리스
	- 배포
	- 롤아웃
- 🔹 효과적인 버전 제어를 위한 브랜치 전략
	- 트렁크 기반 개발 방식과 기능 브랜치 기반 개발 방식
- 🔹 빌드 단계
- 🔹 패키지에 버전을 명시하자.
- 🔹 리소스는 각각 별도로 패키징
- 🔹 릴리스 단계
- 🔹 릴리스를 남의 일로 여기지 말자
- 🔹 패키지를 릴리스 리포지토리로 발행하자.
	- 도커허브, 깃허브 릴리스 페이지, PyPI, 메이븐 센트럴, 비공개형 리포지토리
- 🔹 릴리스는 불변성을 갖게하자.
- 🔹 자주 릴리스하자
- 🔹 릴리스 일정은 투명하게 공유하자.
- 🔹 변경 로그와 릴리스 노트를 발행하자.
- 🔹 배포단계
	- 배포를 자동화하자.
	- 배로는 원자적으로 수행하자.
	- 어플리케이션을 독립적으로 배포하자.
- 🔹 롤아웃 단계
	- 롤아웃을 모니터링하자.
		- 기능 플래그, 서킷 플래그, 다크론치, 카나리배포, 블루-그린 배포등등
	-  기능 플래그를 활용하자.
		- 불리언, 허용목록, 퍼센트 기반 적용, 때론 작은 함수로 구현
	- 서킷 브레이커를 이용해 코드를 보호하자.
	- 서비스 버전은 병렬로 올리자.
	- 다크모드로 론칭하자.
-       참고  : 엠바제인호그빈 웨스트비 팀을 위한 GIT, 제즈 험블과 데이비드 팔리 신뢰할 수 있는 소프트웨어 출시
## 제9장: 긴급대응 온콜 업무
- 🔹 긴급한 비상상황에 대응하는 온콜업무
- 🔹 반드치 샂춰야할 온콜스킬
	- 대응 할 수 있는 가용성
- 🔹 항시 언제라도 대응할 준비를 갖추자.
- 🔹 주의를 늦추지말고 집중하자.
- 🔹 업무 우선순위를 정하자.
	- P1 : 프로덕션 환경의 서비스가 불가능한 상황
	- P2 : 서비스의 사용성이 심각하게 저하된 상황
	- P3 : 서비스의 사용성이 부분적으로 저하된 상황
	- P4 : 서비스 사용성에 문제가 없는 상황
- 🔹 명확하게 의사소통하자.
- 🔹 업무 진척사항을 추적하자.
- 🔹 장애처리의 5가지 단계
	- 선별
	- 조율
	- 완화
	- 해결
	- 후속조치
- 🔹 지원 업무도 엄연한 온콜업무
- 🔹 영웅이 되려하지는 말자
-        참고  : 슬기로운 장애 대응  https://increment.com/on-call/when-the-pager-goes-off/ 구글 사이트 신뢰성 엔지니어링
## 제10장: 견고한 소프트웨어를 위한 기술 설계 절차
- 🔹 고깔형의 기술 설계 절치
- 🔹 올바른 기술 설계를 할려면
	- 문제를 정의하자.
	- 해결방법을 조사하자.
	- 다양한 실험을 해보자.
	- 충분한 시간을  투자하자.
		- 몇시간을 투자
		- 언제 자신의 집중력이 가장 높아지는지를 찾아 일정표에 그 시간 확보
- 🔹 의사소통을 위한 설계 문서 작성 방안
- 🔹 중요한 변경사항을 문서화해두자.
	- 프로젝트에 최소 한달 이상의 엔지니어링 업무가 필요한 경우
	- 변경사항이 소프트웨어의 확장 및 유지보수에 오랫동안 영향을 미칠 경우
	- 변경사항이 다른 팀에 상당한 영향을 미칠 경우
- 🔹 설계문서를 작성하는 이유를 이해하자.
- 🔹 설계문서는 최신상태로 유지하자.
	- 설계 문서의 버전도 관리(위키나 구글문서, 마소워드 등의 형식을 선호)
- 🔹 설계문서템플릿의 기본구조
	- 개요
	- 현재의  상태와 컨텍스트
	- 변경해야 하는 이유
	- 요구사항
		- 사용자 요구사항
		- 기술 요구사항
		- 보안 및 규제 요구사항
		- 기타(중요한 마감일, 예산, 기타등등)
	- 고려할 수 있는 해결책
	- 설계와 아키텍처
		- 시스템 다이어그램
		- UI/UX 변경
		- 코드 변경
		- API변경
		- 영속 계층 변경
		- 테스트 계획
		- 롤아웃 계획
		- 미결 사항
		- 부결
- 🔹 설계과정에서도 협업은 중요
	- 팀의 설계 리뷰 절차를 이해하자.
	- 갑작스런 상황은 만들지 말자.
	- 설계를 논의하며 브레인스토밍을 하자.
	- 설계에 참여하자.
-        참고  : 리처드 힉키 해먹주도 개발, 위페이 엔지니어링 효율적인 소프트웨어 설계문서
## 제11장: 소프트웨어 수명주기 관점의 진화하는 이카텍처 구현
- 🔹 복잡도를 이해하자.
- 🔹 진화하는 아키텍처를 위한 설계 원칙
- 🔹 YAGNI원칙 : 당장 필요치 않다면 구현하지 말 것
- 🔹 최소 충격 원칙: 사용자를 놀래키지 말 것
- 🔹 도메인 지식은 캡슐화 돼야 한다.(도메인주도설계)
- 🔹 진화하는 API를 위한 설계 원칙
	- API 크기는 작게 유지하자.
	- 잘 정의한 서비스 API를 노출하자.
	- API변경에는 호환성을 유지하자.
	- API의 버전을 관리하자.
- 🔹 진화하는 데이터를 위한 설계원칙
- 🔹 데이터베이스를 격리하자.
- 🔹 스키마를 사용하자.
