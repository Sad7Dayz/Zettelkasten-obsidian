## 제4장: 소스코드에 주석 달기
- 🔹 소스 코드 주석 달기의 기본
	- 🔹 주석에 꼭 남겨야 하는 내용
		- 🔹 급하게 수정/대응 했던 부분에 남기는 이슈 트래거 번호(ex. GH-392: 앞 뒤로 오는 공백        문자를 트리밍하도록 보안함)
		- 🔹 다른곳에서 가져온거나 참조한 경우 인용한 곳 (ex. 검사 알고리즘 인용: 사이트)
		- 🔹 파악이 쉽지 않은 알고리즘의 부연 설명
		- 🔹주석에 꼭 남겨야 하는 내용
			- 🔹TODO : 구현이 누락되었거나 나중에 추가 해야 할 경우
			- 🔹FIXME : 수정이 필요한 깨진 코드여서 수정이 필요함
			- 🔹OPTIMIZE :  비효율적인 코드여서 수정이 필요함
			- 🔹HACK :  의심스러운 방식으로 코드를 작성햇거나, 리팩토링이 필요함
			- 🔹NOTE : 다른 개발자가 알아야 할 내용이 있을때 달아둠
			- 🔹REVIEW :  의도한 대로 작동하는지 확인이 필요한 사항이 있음을 나타냄
	- 🔹 명확하고 간결한 주석의 조건
- 🔹 효과적인 주석 남기기
	-  🔹 친절한 주석 기록하기
	-  🔹 능동태로 작성하기
	-  🔹 무엇을 보다는 왜에 집중해서 주석 작성하기
	-  🔹 불필요한 주석 쓰지 않기
	-  🔹 인용, 전제한 소스코드 정확히 표시하기
	-  🔹 이슈트래커와 주석
	-  🔹 주석은 코드를 대신하는것이 아님
-  🔹 코딩 스타일 가이드
	-  🔹 프로그래밍 언어, 회사, 팀별 가이드
		-  🔹 Google 가이드
		-  🔹 GNU 코딩 표준

## 제5장: API
- 🔹 소스 코드 주석 달기의 기본
- 🔹 코드 내 주석과 API 문서 구분하기
	- 🔹 코드내 주석
		- 🔹소스코드 내에서 코드 블록의 로직, 기능 및 목적 등을 직접 설명하여 코드 베이스를 작업     검토하는 개발자를 돕기 위해 사용
		- 🔹코드 안, 버전 관리 시스템에만 쓰임
		- 🔹코드 자체의 개발 및 유지 관리
	- 🔹 API
		- 🔹api를 애플리케이션에 통합하는 개발자를 위해 요청방법, 응답 이해, 오류처리 방법등           api의 기능에 대한 포괄적인 내용을 담음
		- 🔹약속된 규칙에 따라 주석을 작성 해야만 자동 문서 생성 기능 이용가능
		- 🔹개발자가 api를 효과적으로 사용 할 수 있는 외부 가이드 역할
- 🔹 API 개발문서 작성 시 고려할 점
	- 🔹 명확성
		- 🔹 명확하고 이해하기 쉬운 말로 작성
		- 🔹 API 설명시, 육하원칙에 따라 어떤 사용 시나리오에서 쓰이는 것이 바람직한지 설명
	- 🔹 접근성
		- 🔹 필요한 정보를 쉽게 찾을 수 있도록 구성
		- 🔹 단순히 문서 파일로 내용을 제공하는것보다, 적절한 문서화 도구를 사용하는것이 좋음.
			- 🔹 API문서를 자동화 도구로 제공
	- 🔹 완전성
		-  🔹 모든 API 엔드포인트, 파라미터, 응답 타입 및 에러 코드에 대한 설명이 포함
		-  🔹 실제로 실행 되는 코드와 동작이 일치(예:[라인레퍼런스](https://developers.line.biz/en/reference/messaging-api/#get-number-of-reply-messages)
 )
		- 🔹  CURL명령어나 API플레이그라운드 예시 제공 (예: [라인 메세지](https://developers.line.biz/en/docs/messaging-api/sending-messages) )
	- 🔹  유지관리
		-  🔹 업데이트될 때마다 문서도 함께 업데이트
		-  🔹 API문서를  코드와 같은수준으로 리뷰-테스트-출시하는 프로세스 정립
	- 🔹  피드백 수집
		- 🔹 사용자 피드백을 수집하고 이를 바탕으로 문서를 지속해서 개선하는 활동
		- 🔹 피드백 수집방법(별점/댓글, 조회수, 데이터 애널리틱스등)
- 🔹 API 문서 작성해보기
	- 🔹  비지니스 요구 사항 이해하기
		- 🔹 비지니스 요구사항 도출
			- 🔹 어떤 배경속에서 그 API가 개발되었는지 확인
			- 🔹 API를 사용할 사람들의 상황, 행동, 어려움 
			- 🔹 육하원칙에 따라 정리해보기
		- 🔹 도메인 지식 구축하기
			- 🔹 식당과 관련된 도메인 지식
				- 🔹 식당 내 운영체계, 메뉴 구성, 가게 운영 방식 등을 통해 식당 영업에 대한             지식습득
		- 🔹자료 구조 이해
			- 🔹  가장 핵심이 되는 자료 구조
				- 🔹  레코드 : 데이터베이스에 저장하는 개별 레코드의 논리적인 개념을 정의하는         자료구조
				- 🔹 뷰모델 : 웹이나 사용자  UI에 표시될 화면상의 데이터를 정의 하는 자료                구조와 메커니즘
				- 🔹  모델 : api를 통해 들어오는 요청이나 api가 내보내는 응답에서 표현하는              자료구조
		- 🔹 사용자 인증과 권한 체계 이해하기(RBAC같은 인증과 권한)
			- 🔹 인증과 권한 부여
			- 🔹  인증과 권한 체계 내용을 고객/개발자가 잘 이해 할 수 있도록 문서로 정확히 설명
		- 🔹 예외 상황 정리와 문제 해결 방안 제시하기
			- 🔹 API를 제공할 때 명확한 오류 코드를 정하는 것이 좋음
				- 🔹 오류의 종유를 알 수 있게 하는 것 외에도 , 오류가 어느 위치에서 발생한 것인지           구분 할 수 있게 상화별로 별도 코드 부여
			- 🔹 오류코드를 세분화하면 좋은점
				- 🔹 오류가 왜 발생하고 어디를 원인으로 봐야 할지 알 수 있는 힌트 제공
			- 🔹 오류 코드를 API문서의 부록으로 정리하여 제공하면, 다른개발자들에게 많은               도움을 줄 수 있음
				- 🔹 오류 코드표와 각 오류코드의 발생 원인, 해결방안을 정리 해두면 유익한              내용이 됨
		-  🔹 오류코드를 정의할때 체계성을 만드는 방법
			- 🔹 세자리 숫자에서 앞자리 숫자는 대전제, 나머지 두자리는 상세내용을 담으면 쉽게         의미 체계를 만들 수 있음.
			- 🔹 오류코드의 의미를 어떻게 파악 할 수 있는지 상세한 방법과 각 코드의 의미를              문서화 해두면 유익함.
			- 🔹 각 엔드포인트의 사용 예시를 담은 샘플코드
		-  🔹 API 문서화에 담길 내용 정리 하기
			- 🔹 API를 사용 할 수 있는 방법과 보안 고려 사항, API 변경 이력은 별도의 문서로              작성하여 쉽게 찾아 볼 수 있도록 같이 배포(예시 :[swagger](https://editor.swagger.io)  )
