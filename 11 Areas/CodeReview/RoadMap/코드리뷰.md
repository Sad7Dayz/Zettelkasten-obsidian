# 코드 리뷰

## 팀 전체 (Team Wide Practices)

- [ ] 코드 리뷰 프로세스를 문서화하고 표준화합니다. 
- [ ] 코드 리뷰의 목적이 모든 사람에게 명확하도록 합니다. 
- [ ] "완료의 정의(Definition of Done)"가 문서화되고 모든 사람에게 명확하도록 합니다. 
- [ ] 팀원들이 코드 리뷰에 참여하도록 장려합니다. 
- [ ] 코드 리뷰에서의 의견 충돌 해결을 위한 프로세스를 정의합니다.
- [ ] 스타일 선호도에 대한 명확한 스타일 가이드를 마련합니다. 
- [ ] 린팅(linting), 스니핑(sniffing) 등 자동화를 사용하여 코드 리뷰 속도를 높입니다. 
- [ ] 코드 리뷰 처리 시간에 대한 명확한 기대치를 설정합니다. 
- [ ] 코드 리뷰를 위한 충분한 시간을 제공하고 우선순위로 삼도록 합니다. 
- [ ] 코드 리뷰를 지식 공유와 학습의 기회로 활용합니다. 
- [ ] 여러 기능 분야에 걸친 지식 함양을 위해 잘 모르는 영역의 코드 리뷰를 장려합니다. 
- [ ] 코드 리뷰 프로세스를 지속적으로 모니터링하고 개선합니다. 
- [ ] 양질의 피드백을 꾸준히 제공한 사람들에게 인정과 보상을 합니다. 
- [ ] 코드 리뷰를 일방적인 프로세스로 취급하지 않고, 소통과 협업을 장려합니다. 

---

## 모두 (Everyone)

- [ ] 정기적인 코드 리뷰 세션을 열어 리뷰 과정에서 발생하는 광범위한 트렌드나 문제에 대해 논의합니다. 
- [ ] 공식적인 코드 리뷰를 제출하기 전에 개발 중에 피드백을 구하도록 장려합니다. 
---

### 작성자 (Author)

#### 개발 중 (During Development)

- [ ] 코딩 표준 및 기타 모든 팀 가이드라인을 따릅니다. 
- [ ] 전반적인 프로젝트 설계 및 아키텍처와 일관성을 유지합니다. 
- [ ] 변경 사항이 버그 수정인 경우, 실패하는 테스트를 작성합니다. 
- [ ] 복잡한 작업을 더 작고 관리하기 쉬운 PR(Pull Request)로 나눕니다. 
- [ ] 변경 사항이 시스템의 다른 부분에 미치는 영향을 고려합니다. 
- [ ] 변경 사항에 대한 질문이나 우려 사항을 기록하여 리뷰 중에 논의할 수 있도록 합니다. 
- [ ] 자동화된 테스트를 작성합니다. 
- [ ] 필요한 경우 기능 또는 변경 사항에 대한 문서를 작성합니다. 
- [ ] 변경으로 인해 더 이상 유효하지 않게 된 문서를 업데이트합니다. 

#### 개발 후 (After the Development)

- [ ] 리뷰를 위해 제출하기 전에 자신의 코드를 리뷰합니다. 
- [ ] 필요한 모든 테스트와 문서를 포함하여 변경 사항이 완전하고 리뷰 준비가 되었는지 확인합니다. 
- [ ] 개발 환경에서 코드 변경 사항이 적절하게 테스트되었는지 확인합니다. 
- [ ] 코드가 프로젝트의 코딩 표준과 모범 사례를 준수하는지 다시 확인합니다. 
- [ ] 잠재적인 성능, 보안 또는 확장성 문제를 파악하고 리뷰 중 논의할 수 있도록 기록해 둡니다. 
- [ ] PR에 적절한 제목, 설명, 스크린샷, 관련 링크, 구성 변경 사항 등을 추가해야 합니다. 
- [ ] 열린 마음으로 리뷰 프로세스에 접근하고, 다른 팀원들로부터 배우고 협력하려는 의지를 갖습니다. 

#### 코드 리뷰 후 (After the Code Review)

- [ ] 제기된 우려 사항이나 질문을 포함하여 받은 모든 피드백을 처리합니다. 
- [ ] 제안된 변경 사항을 구현하고 필요한 경우 설명을 제공합니다.
- [ ] 변경 후 테스트를 실행하여 모두 통과하는지 확인합니다. 
- [ ] 변경 사항의 영향을 받는 모든 문서나 코드 주석을 업데이트합니다. 
- [ ] 변경 사항에 대해 확신이 서지 않으면 다른 팀원에게 피드백을 구합니다. 
- [ ] 필요한 경우 2차 리뷰를 위해 업데이트된 코드를 제출합니다. 

---

### 리뷰어 (Reviewer)

#### 리뷰 전 (Before Reviewing)

- [ ] 변경이 이루어진 요구 사항과 맥락을 이해합니다.
- [ ] 요구 사항을 바탕으로 변경 사항에 포함되어야 할 항목 목록을 준비합니다.
- [ ] 코드베이스와 그 아키텍처를 이해하고 있는지 확인합니다.
- [ ] 변경과 관련된 모든 문서나 설계 사양을 검토합니다.
- [ ] 변경으로 인해 발생할 수 있는 잠재적 위험이나 문제 목록을 만듭니다.
- [ ] 코드 변경의 범위와 영향에 따라 필요한 리뷰 수준을 결정합니다.
- [ ] 열린 마음으로 프로세스에 접근합니다. 
- [ ] 건설적인 피드백을 제공하고 코드 품질을 개선하기 위해 협력할 의지를 갖습니다.

#### 코드 리뷰 중 (During Code Review)

- [ ] 가독성, 유지보수성, 확장성을 포함하여 코드의 전반적인 품질을 고려합니다.
- [ ] 리뷰 과정에서 발생하는 모든 문제나 우려 사항을 해결하기 위해 작성자와 기꺼이 협력합니다.
- [ ] 피드백을 할 때 존중하고 전문적인 태도를 유지하며, 인신공격이나 비하하는 발언을 피합니다.
- [ ] 개선을 위한 구체적인 제안과 우려 사항에 대한 설명을 포함하여 명확하고 실행 가능한 피드백을 제공합니다. 
- [ ] 잠재적인 성능, 보안 또는 확장성 문제를 파악하고 작성자와 논의합니다.
- [ ] 가장 중요한 문제에 먼저 초점을 맞춰 피드백의 우선순위를 정합니다.
- [ ] 코드 변경에 포함된 모든 테스트를 검토하여 기능과 엣지 케이스를 적절하게 다루는지 확인합니다. 
- [ ] 코드 변경이 프로젝트의 코딩 표준과 모범 사례를 준수하는지 확인합니다.
- [ ] 관련 문서가 업데이트되었는지 확인합니다. 
- [ ] 스타일링에 대해서는 팀 전체의 스타일 가이드가 절대적인 기준입니다. 개인적인 선호도 대신 해당 가이드에 따라 변경 사항을 확인합니다.
- [ ] 개선을 제안하는 코멘트를 남기되, 표준을 충족하는 데 중요하지 않은 경우 "Nit"(사소한 지적)라는 접두사를 붙입니다.
- [ ] 완벽함이 아닌 지속적인 개선을 추구합니다.
- [ ] 단기 및 장기적인 고려 사항을 염두에 둡니다.
- [ ] 코드 리뷰의 대안 또는 보완책으로 페어 프로그래밍 사용을 고려합니다.
- [ ] 좋은 습관을 강화하고 팀 사기를 높이기 위해 건설적인 비판 외에 긍정적인 피드백도 제공합니다.

#### 코드 리뷰 후 (After the Code Review)

- [ ] 상충하는 의견을 시기적절하게 해결합니다.
- [ ] 의견 불일치로 인해 PR이 방치되지 않도록 합니다.
- [ ] 작성자가 모든 피드백을 처리했는지 확인합니다. 
- [ ] 업데이트된 코드를 검토하고 제안된 변경 사항이 예상대로 구현되었는지 확인합니다. 
- [ ] 테스트를 다시 실행하여 모두 통과하는지 확인합니다.
- [ ] 작성자가 가질 수 있는 모든 질문이나 우려 사항을 처리합니다. 
- [ ] 작성자의 피드백에 개방적이며 필요한 경우 자신의 피드백을 조정할 의지를 갖습니다. 

---

## 승인 후 (작성자/리뷰어)

- [ ] 승인된 코드 변경 사항을 메인/피처 브랜치에 병합합니다. 
- [ ] 프로덕션 환경에서 코드 변경이 예상대로 작동하는지 확인합니다. 
- [ ] 코드 변경의 성능과 기능을 모니터링하고 발생하는 모든 문제를 해결합니다. 
- [ ] 코드 변경의 성공적인 완료를 축하합니다!
